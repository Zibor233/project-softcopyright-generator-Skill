# Project SoftCopyright Generator

## Overview

本技能用于读取本地已有项目代码，自动分析项目结构、功能模块、技术架构与关键源码，并生成适合软件著作权登记使用的说明资料与源代码资料。

适用对象：

- 已经有本地项目代码，希望直接整理成软著资料
- 需要先读取项目，再由 AI 给出软著名称、简介、模块划分和材料建议
- 需要按软著常见要求抽取代码文档：前 30 页与后 30 页，每页 60 行，且代码片段连续
- 需要在正式生成前让 AI 先给建议，再由用户确认是否采用

## Output Structure

建议统一在单个工作目录下生成资料，例如：

```text
<项目根目录>/
└─ 软著资料/
   ├─ 草稿/
   │  ├─ 软著建议稿.md
   │  ├─ 申请表信息.md
   │  ├─ 代码文件候选清单.md
   │  ├─ 代码文件选择.json
   │  ├─ xxx_系统说明文档.md
   │  ├─ xxx_源代码文档.md
   │  └─ 生成报告.md
   └─ 正式资料/
      ├─ xxx_系统说明文档.docx
      ├─ xxx_源代码文档.docx
      └─ xxx_软著申请信息.txt
```

规则要求：

- 候选清单、待确认内容、申请表建议稿写入 `草稿/`
- 系统说明文档和源代码文档先生成 `草稿/*.md`，再转换为 `正式资料/*.docx`
- 申请表正式稿直接输出到 `正式资料/xxx_软著申请信息.txt`
- `正式资料/` 目录只保留 3 个正式文件，不保留 md、报告或额外脚本
- 正式文档中的软件名称和版本号应以已确认字段为准

## Template Files

当前技能目录中的模板文件建议按以下顺序使用：

1. `templates/run-project-softcopyright-generator-template.py`
   - 用于串联整套流程的总入口脚本
   - 首次运行时会先生成 `草稿/代码文件候选清单.md` 和 `草稿/代码文件选择.json`
   - 当 `代码文件选择.json` 已确认并执行“继续生成正式资料”后，再继续生成申请表 txt、草稿 md 和正式 docx

2. `templates/propose-code-selection-template.py`
   - 用于扫描项目源码，生成 `草稿/代码文件候选清单.md` 和 `草稿/代码文件选择.json`
   - 适用于正式代码抽取前的候选整理与人工确认阶段

3. `templates/generate-application-info-template.py`
   - 用于根据已确认的软著口径生成 `草稿/申请表信息.md`
   - 同时输出 `正式资料/xxx_软著申请信息.txt`

4. `templates/generate-doc-template.py`
   - 用于根据项目分析结果和已确认的软著口径生成 `草稿/xxx_系统说明文档.md`
   - 默认在每个功能模块下保留一行 `【图片预留截图位置】`

5. `templates/generate-source-template.py`
   - 用于根据已确认的 `草稿/代码文件选择.json` 生成 `草稿/xxx_源代码文档.md`
   - 按前 30 页、后 30 页、每页 60 行的规则组织代码页内容

6. `templates/render-print-docx-template.py`
   - 用于将 `草稿/xxx_系统说明文档.md` 和 `草稿/xxx_源代码文档.md` 转换为 `正式资料/*.docx`
   - 申请表不转换 docx，正式资料中只保留 `.txt`

配套参考文件：

- `references/application-fields.md`：申请表字段口径与一致性要求
- `references/design-spec.md`：项目分析、代码抽取、文档输出的总体参考规范

统一配置建议：

- `system_name`
- `short_name`
- `version`
- `project_root`
- `workdir`
- `draft_dir_name`
- `final_dir_name`

## When to Use

- 用户已经有本地项目目录，希望 AI 自动读取代码后生成软著材料
- 用户希望从现有代码仓库中整理系统说明文档、源代码文档、申请表素材
- 用户明确提到“根据项目代码生成软著资料”“根据本地程序自动写说明文档”
- 用户希望 AI 先分析项目并给出软著名称、系统简介、模块建议，再确认后生成

## Core Principles

1. **先读项目，后写资料**

必须优先读取用户提供的本地项目目录，理解真实代码结构、技术栈、模块边界、入口文件、核心功能、数据流和部署方式，禁止在未读取代码前直接虚构系统说明。

2. **先建议，后确认**

读取项目后，先输出一版“软著材料建议稿”，至少包括：

- 建议的软件全称
- 建议的软件简称
- 建议的版本号
- 建议的软件简介
- 建议的开发目的
- 建议的主要功能模块
- 建议的技术特点
- 建议的运行环境和开发工具

然后明确询问用户：

> “是否使用该软著介绍与资料框架继续生成下一步文档？”

在用户明确确认前，不进入正式文档生成阶段。

3. **以代码为准，不以想象为准**

所有说明内容必须尽量锚定实际代码，包括但不限于：

- 项目目录结构
- 入口文件
- 核心业务模块
- 数据模型或数据库访问层
- 接口、服务、路由或控制器
- 前端页面、组件或交互逻辑
- 配置文件、依赖清单、脚本命令

4. **代码文档必须满足分页抽取规则**

源代码文档需要按照以下规则组织：

- 总体目标为 60 页代码
- 前 30 页：从项目的前部核心代码中选取连续代码
- 后 30 页：从项目的后部核心代码中选取连续代码
- 每页按 60 行计算
- 前 30 页和后 30 页应分别来自两段连续的代码区间
- 两段代码区间应尽量来自不同阶段或不同模块，避免高度重复
- 代码应尽量覆盖项目核心价值，而非仅选配置文件或样板代码

## Standard Workflow

```text
输入：本地项目目录 / 项目文件
  │
  ▼ 步骤1：扫描项目并读取关键代码
  │
  ├─ 识别技术栈、目录结构、入口文件、核心模块
  ├─ 过滤无效目录（依赖、构建产物、缓存等）
  ├─ 提取系统功能、业务流程、技术亮点
  └─ 估算可用于软著代码抽取的核心源码范围
  │
  ▼ 步骤2：输出软著建议稿
  │
  ├─ 生成软件全称、简称、版本号建议
  ├─ 生成功能简介、开发目的、技术特点
  ├─ 给出模块划分和文档目录建议
  └─ 询问用户是否采用该介绍与框架
  │
  ▼ 步骤3：等待用户确认
  │
  ├─ 若用户要求修改，则先调整建议稿
  └─ 若用户确认通过，再进入正式生成
  │
  ▼ 步骤4：生成代码候选清单
  │
  ├─ 生成代码文件候选清单
  ├─ 生成可编辑的代码文件选择 JSON
  ├─ 由模型填写选择理由、起止行和是否选用
  └─ 再次询问用户是否采用该代码抽取方案
  │
  ▼ 步骤5：规划代码抽取区间
  │
  ├─ 明确前30页使用哪一段连续代码
  ├─ 明确后30页使用哪一段连续代码
  ├─ 每页按60行换页
  └─ 尽量覆盖入口、业务、数据、界面等核心逻辑
  │
  ▼ 步骤6：生成系统说明文档
  │
  ├─ 根据真实代码生成系统概述
  ├─ 生成功能模块说明、架构说明、运行环境说明
  └─ 输出适合软著登记使用的说明文档
  │
  ▼ 步骤7：生成源代码文档
  │
  ├─ 按“前30页 + 后30页”规则编排
  ├─ 保持代码连续性与原始格式
  └─ 输出适合软著提交的源代码文档
```

## Step 1 — 扫描本地项目

### 必须读取的内容

- 项目根目录结构
- 依赖声明文件，如 `package.json`、`requirements.txt`、`pom.xml`、`go.mod`、`Cargo.toml`
- 入口文件，如 `main.py`、`app.py`、`main.js`、`server.js`、`src/main.ts`、`Program.cs`
- 核心业务目录，如 `src/`、`app/`、`server/`、`backend/`、`frontend/`、`modules/`
- 配置与环境文件，如 `.env.example`、`config/`、`settings.py`
- 数据层文件，如 `models/`、`schemas/`、`migrations/`、`sql/`
- 路由、接口、服务、控制器、组件等核心实现

### 默认忽略的目录

- `node_modules/`
- `dist/`
- `build/`
- `.git/`
- `.idea/`
- `.vscode/`
- `.next/`
- `coverage/`
- `tmp/`
- `logs/`
- 二进制文件、压缩包、图片资源、字体文件、视频文件

### 扫描目标

扫描后至少要提炼出以下内容：

- 项目属于什么类型的软件
- 系统面向什么用户或业务场景
- 项目的前后端结构或单体结构
- 核心功能模块有哪些
- 哪些文件最能代表系统的原创性与功能完整性
- 哪些代码适合放入软著代码文档

## Step 2 — 先输出建议，再征求确认

读取项目后，先给用户一份结构化建议稿，格式建议如下：

```markdown
## 软著建议稿

- 软件全称：
- 软件简称：
- 版本号：
- 软件分类：
- 开发完成日期建议：
- 运行环境：
- 开发工具：
- 编程语言：
- 源程序量预估：
- 开发目的：
- 面向领域：
- 主要功能：
- 技术特点：
- 核心模块：
```

然后必须追加一个确认问题，表达要明确，例如：

```text
以上是根据当前项目代码自动整理出的软著介绍建议。
请确认是否采用该版本继续生成正式资料；如需修改，我会先按你的意见调整，再进入下一步。
```

### 交互规则

- 用户未确认前，不生成正式版说明文档与源代码文档
- 用户要求修改名称、功能描述、技术特点时，先重写建议稿
- 用户确认后，再锁定本次生成所使用的软著信息

## Step 3 — 代码候选清单与选择文件

在正式抽取代码前，先生成两个草稿文件：

- `草稿/代码文件候选清单.md`
- `草稿/代码文件选择.json`

其中 `代码文件选择.json` 至少应包含以下字段：

- `selected`
- `start_line`
- `end_line`
- `model_reason`

执行规则：

- 脚本只负责列出候选文件，不直接默认抽取全库
- 模型需要结合项目业务、入口文件、核心模块和业务链路填写选择结果
- 用户需要确认抽取文件和行段后，才能进入正式代码抽取
- 代码文档必须以已确认的选择文件为准
- 当用户明确回复“确认，继续生成正式资料”后，必须直接生成正式资料目录中的最终文件，而不是只停留在中间草稿

## Step 4 — 代码抽取规则

### 总体规则

- 代码文档按 60 页组织
- 每页 60 行
- 总行数目标为 3600 行
- 前 30 页对应前半段连续代码，共 1800 行
- 后 30 页对应后半段连续代码，共 1800 行

### 前 30 页选取规则

优先从以下代码中选择一段连续区间：

- 项目入口与初始化代码
- 路由或接口注册逻辑
- 核心业务服务
- 数据模型与关键算法
- 能体现系统主体价值的前部源码

要求：

- 尽量保证来自真实连续文件顺序或同一业务链路
- 不要频繁跳文件拼接成碎片化代码
- 不要只选空壳模板、纯样式文件、纯常量文件

### 后 30 页选取规则

优先从以下代码中选择另一段连续区间：

- 后续核心业务模块
- 重要前端页面或组件逻辑
- 数据处理、报表、审核、工作流、消息、权限等模块
- 能补足前 30 页不足部分的关键源码

要求：

- 与前 30 页相比，尽量体现不同模块或不同层次的实现
- 同样必须保持连续性
- 如果项目规模不足 3600 行，则优先保持连续与真实性，不得强行编造代码

### 连续代码的判定原则

以下方式视为优先：

- 同一文件内连续行号区间
- 同一目录下按业务链路连续排列的多个文件
- 按入口 -> 路由 -> 服务 -> 数据层的自然调用链顺序组织

以下方式尽量避免：

- 每个文件只截几行后再跳到另一个文件
- 只摘抄注释、类型定义、配置项而缺少业务实现
- 大量重复模板代码或自动生成代码

## Step 5 — 系统说明文档生成要求

系统说明文档应基于项目真实代码生成，建议至少包含以下章节：

1. 软件概述
2. 开发背景与目的
3. 运行环境
4. 开发工具与技术栈
5. 系统总体架构
6. 功能模块说明
7. 关键业务流程说明
8. 数据设计或核心数据对象说明
9. 主要技术特点
10. 总结

### 写作要求

- 用中文正式表达，适合软著材料场景
- 内容与实际代码一致，避免夸张营销式表述
- 描述模块时尽量对应真实目录、页面、接口或服务
- 如果项目包含前后端，说明前端负责展示交互，后端负责业务与数据处理
- 如果项目是单体应用，则说明其模块边界与数据处理流程

## Step 6 — 源代码文档生成要求

### 文档结构

1. 封面
2. 目录
3. 源代码正文

### 正文要求

- 只放代码，不插入大段中文功能说明
- 可以保留必要的文件标题，例如：`3.1 main.py 源代码`
- 代码保持原始缩进、换行和大小写
- 代码字体建议为 `Courier New`
- 文件标题按顺序编号，便于审查

### 分页要求

- 严格按每页 60 行控制
- 前 30 页使用第一段连续代码
- 后 30 页使用第二段连续代码
- 若代码块跨页，允许自然续接到下一页
- 若不足 60 页，应如实说明项目代码规模不足，并优先保证真实性

## Mandatory Constraints

- 不得在未读取项目代码前直接生成完整软著介绍
- 不得跳过“先建议、再确认”的交互步骤
- 不得跳过代码候选清单和代码选择确认步骤直接抽全量代码库
- 不得在正式资料生成阶段临时新建额外的排版 Python 脚本，必须优先复用 `templates/` 下的内置模板
- 不得为了凑满页数而编造不存在的业务代码
- 不得脱离真实源码分析去虚构系统说明或代码材料
