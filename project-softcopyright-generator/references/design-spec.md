# Project SoftCopyright Generator 参考规范

本文件用于约束基于真实项目代码生成软著资料时的分析、确认、抽取和输出规则。

## 目标定位

本 skill 的目标是读取本地真实项目代码，自动整理软著资料草稿，并在用户确认后输出正式文档。

核心产出包括：

- 软著建议稿
- 申请表信息草稿
- 代码文件候选清单
- 代码文件选择 JSON
- 系统说明文档
- 源代码文档
- 软著申请信息 TXT
- 生成报告

## 推荐目录结构

```text
<项目根目录>/
└─ 软著资料/
   ├─ 草稿/
   │  ├─ 软著建议稿.md
   │  ├─ 申请表信息.md
   │  ├─ 代码文件候选清单.md
   │  ├─ 代码文件选择.json
   │  ├─ xxx_系统说明文档.md
   │  ├─ xxx_源代码文档.md
   │  └─ 生成报告.md
   └─ 正式资料/
      ├─ xxx_系统说明文档.docx
      ├─ xxx_源代码文档.docx
      └─ xxx_软著申请信息.txt
```

## 模板入口建议

建议优先使用以下模板文件：

1. `templates/run-project-softcopyright-generator-template.py`
2. `templates/propose-code-selection-template.py`
3. `templates/generate-application-info-template.py`
4. `templates/generate-doc-template.py`
5. `templates/generate-source-template.py`
6. `templates/render-print-docx-template.py`

推荐统一配置字段：

- `system_name`
- `short_name`
- `version`
- `project_root`
- `workdir`
- `draft_dir_name`
- `final_dir_name`

## 项目分析优先级

分析项目时，优先读取以下内容：

1. 项目根目录结构
2. 依赖声明文件，如 `package.json`、`requirements.txt`、`pom.xml`、`go.mod`
3. 入口文件，如 `main.py`、`app.py`、`main.js`、`server.js`、`src/main.ts`
4. 路由、控制器、接口、服务、模型、组件、页面等核心实现
5. README、产品说明、需求文档、架构文档
6. 配置文件和运行脚本

默认忽略：

- `node_modules/`
- `dist/`
- `build/`
- `.git/`
- `.idea/`
- `.vscode/`
- `.next/`
- `coverage/`
- 图片、视频、字体、压缩包、二进制文件

## 业务理解规则

申请表信息和系统说明文档不能只根据代码目录泛泛生成，必须先理解软件业务。

至少要形成以下判断：

- 项目属于什么软件类型
- 软件面向什么行业或场景
- 主要用户是谁
- 核心价值是什么
- 主要功能模块有哪些
- 哪些功能最适合写入软著资料
- 哪些代码最能体现软件真实功能和运行逻辑

如果项目材料不足，应先让用户补充项目背景、产品说明或目标场景，而不是直接生成空泛描述。

## 建议稿确认规则

读取项目后，先输出建议稿，再询问用户是否采用。

建议稿至少包括：

- 软件全称
- 软件简称
- 版本号
- 软件分类
- 开发目的
- 面向领域 / 行业
- 主要功能
- 技术特点
- 运行环境
- 开发工具
- 编程语言

在用户明确确认前，不得直接生成正式文档。

## 申请表字段规范

申请表字段口径应参考 `application-fields.md`，至少遵守以下要求：

- 软件全称必须由用户确认
- 版本号必须由用户确认
- 著作权人、日期、硬件环境、操作系统等字段必须允许用户补充或修改
- 开发工具只写 IDE 或编辑器，不写框架名
- 运行支撑环境填写项目运行依赖的软件环境，如 `Node.js`、`Python`、数据库、中间件、浏览器等
- 正式资料中的软件名称和版本号必须与申请表信息一致

## 代码候选选择规则

代码抽取前，必须先生成：

- `草稿/代码文件候选清单.md`
- `草稿/代码文件选择.json`

`代码文件选择.json` 中至少应包含：

- `path`
- `selected`
- `start_line`
- `end_line`
- `model_reason`

执行要求：

- 候选清单只列证据，不直接默认抽取全库
- 模型需先结合项目业务、代码职责、入口结构和功能模块填写选择结果
- 用户确认后，代码文档才可按该选择结果正式生成
- 抽取脚本必须以确认后的选择文件为准
- 当用户明确回复“确认，继续生成正式资料”后，应直接生成正式资料目录中的最终文件

## 代码分页规则

源代码文档按以下规则组织：

- 每页 60 行
- 总体目标为 60 页
- 前 30 页使用第一段连续代码
- 后 30 页使用第二段连续代码
- 两段代码区间尽量来自不同模块或不同层次
- 代码必须保持真实性和连续性

优先选取：

- 入口代码
- 路由或接口注册逻辑
- 核心服务
- 数据模型
- 核心业务页面或组件
- 报表、审核、权限、工作流、数据处理等关键模块

避免选取：

- 自动生成代码
- 无业务含义的样板文件
- 纯样式文件
- 过短且无实际业务价值的配置文件
- 只抽零散几行后频繁跳文件的碎片化组合

## 系统说明文档章节建议

系统说明文档建议至少包含：

1. 软件概述
2. 开发背景与目的
3. 运行环境
4. 开发工具与编程语言
5. 系统总体架构
6. 功能模块说明
7. 关键业务流程说明
8. 数据设计说明
9. 主要技术特点
10. 总结

写作要求：

- 用中文正式表达
- 内容必须基于真实项目
- 尽量对应真实目录、模块、页面、接口和服务
- 避免空泛营销化表述

## 正式资料输出规则

正式资料建议输出到 `正式资料/` 目录，包括：

- `xxx_软著申请信息.txt`
- `xxx_系统说明文档.docx`
- `xxx_源代码文档.docx`

草稿目录建议保留：

- `xxx_系统说明文档.md`
- `xxx_源代码文档.md`
- `生成报告.md`

生成报告中建议记录：

- 软件名称
- 版本号
- 输出文件列表
- 代码文档前后两段代码的行数
- 是否仍存在待用户确认字段
- 是否存在页数不足或补充抽取的情况

## 最终约束

- 不得在未读取项目代码前直接生成完整软著介绍
- 不得跳过“先建议、再确认”的交互步骤
- 不得跳过代码候选清单和代码选择确认步骤直接抽取全量代码库
- 不得在正式资料生成阶段临时编写新的排版脚本，必须优先使用内置模板
- 不得为了凑页数而编造不存在的业务代码
- 不得脱离真实源码分析去虚构系统说明、申请表口径或代码材料
